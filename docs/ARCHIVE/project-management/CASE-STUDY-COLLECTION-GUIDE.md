## 📋 收集目标

本文档提供案例收集的标准化指南，确保案例的真实性、完整性和可验证性。

**收集原则**：

1. **真实性**：所有案例必须有明确的来源和验证
2. **完整性**：案例描述应包含关键信息（规模、性能、技术栈等）
3. **可验证性**：案例应可追溯和验证
4. **多样性**：覆盖不同行业和应用场景

## 2. 案例收集方法

**收集来源**：

3. **官方文档**：
   - 案例研究文档
   - 白皮书
   - 技术报告

**收集步骤**：

1. **搜索官方资源**：

3. **记录案例信息**：
   - 记录案例来源
   - 记录案例描述
   - 标注验证状态


## 4. 案例验证标准

### 4.1 验证标准

**案例验证标准**：

1. **来源可靠性**：

   - ✅ 官方技术会议演讲
   - ✅ 官方博客文章
   - ✅ 经过同行评审的论文
   - ✅ 合作伙伴官方案例
   - ⚠️ 社区博客和讨论（需要验证）

2. **信息完整性**：

   - ✅ 完整的案例描述
   - ✅ 明确的技术栈
   - ✅ 关键指标数据
   - ✅ 实施步骤和经验总结
   - ⚠️ 部分信息缺失（需要补充）

3. **可追溯性**：
   - ✅ 可追溯的来源链接
   - ✅ 可验证的技术信息
   - ✅ 可联系的联系人
   - ⚠️ 来源不明确（需要验证）

### 4.2 验证状态

**验证状态分类**：

- ✅ **已验证**：案例来源可靠，信息完整，可追溯
- ⚠️ **待验证**：案例来源需要验证，信息不完整
- ❌ **已过期**：案例已过时，不再适用
- 🔄 **计划中**：计划收集案例，尚未完成

## 6. 案例标注规范

### 6.1 标注格式

**标准标注格式**：

```markdown
> **⚠️ 来源标注说明**：本节案例均已标注来源和验证状态。

| 案例         | 行业 | 场景 | 规模 | 性能 | 来源 | 验证状态  | 备注 |
| ------------ | ---- | ---- | ---- | ---- | ---- | --------- | ---- |
| **案例名称** | 行业 | 场景 | 规模 | 性能 | 来源 | ✅ 已验证 | 备注 |
```

### 6.2 标注要求

**标注要求**：

1. **案例来源**：明确标注案例来源（技术会议、官方博客、社区资源、合作伙伴）
2. **验证状态**：明确标注验证状态（已验证/待验证/已过期）
3. **收集日期**：记录案例收集日期
4. **备注信息**：添加必要的备注信息（验证说明、补充信息等）

## 8. 案例质量保证

### 8.1 质量检查清单

**案例质量检查**：

- [ ] 案例来源是否可靠
- [ ] 案例描述是否完整
- [ ] 技术栈是否明确
- [ ] 关键指标是否准确

### 8.2 案例对比验证

**对比验证方法**：

1. **多源对比**：

   - 对比多个案例来源
   - 验证案例一致性
   - 识别异常案例

2. **历史对比**：

   - 对比历史案例
   - 验证案例趋势
   - 识别案例异常

3. **理论验证**：
   - 对比理论预期
   - 验证案例合理性
   - 识别案例错误

### 9.2 收集脚本

**示例脚本**：

```bash
#!/bin/bash
# 案例收集脚本

# 配置
CASE_DIR="cases"
TEMPLATE_FILE="case-template.md"

# 创建案例目录
mkdir -p "$CASE_DIR"

# 创建案例文件
case_name="$1"
case_file="$CASE_DIR/$case_name.md"

# 复制模板
cp "$TEMPLATE_FILE" "$case_file"

# 提示
echo "案例文件已创建: $case_file"
echo "请填写案例信息"
```
